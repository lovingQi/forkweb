# 日志诊断与回放工具开发现状

本文记录 forkweb 日志诊断与回放工具的设计结论、当前实现状态、测试结果、使用方式、已知问题和后续计划。

## 研究结论

- 现有 forkweb 通过 `/ws/high` 接收 `pose`、`vel`、`laser_data`、`path_points`、`clearances`、`robot_size`。
- 现有 forkweb 通过 `/ws/low` 接收 `status`、`battery`、`score`、`alarm`、`current_routes`、`fork_info`、IO 等状态。
- 日志中 `FltStatus: RmstoSendArg` 可提供离线回放主数据源，包括位置、朝向、电量、定位分、任务 ID、错误、货叉高度等。
- 日志中 `InfoStatus: ROBOT STATUS` 可补充状态、速度、电机、充电、定位分等。
- 日志中 `error_name,error_str=ERRORxxxx,{...}` 可建立错误码字典，正文中的 `ERRORxxxx` 可建立错误发生点。

## 已选方案

- 直接集成到 forkweb。
- 新增 `/replay` 页面，不改动现有实时监控首页。
- 使用 Node.js + TypeScript 本地辅助服务处理日志目录、地图文件、解析、回放和报告。
- 第一版本地优先，后续预留车端部署。
- 地图选择支持地图目录自动加载，失败时可手动填写地图文件。
- 报告导出支持 Markdown 和 JSON。

## 当前架构

```text
forkweb
├── replay-server/              # 本地日志诊断服务
│   ├── src/index.ts            # HTTP + WebSocket 入口
│   ├── src/parser/             # 日志、状态、错误码、任务解析
│   └── src/core/               # session、timeline、噪声折叠、根因、缓存、诊断包、报告生成
├── src/views/Replay.vue        # 日志诊断页面
├── src/api/replay.ts           # 诊断 API 封装
├── src/stores/replay.ts        # 诊断状态
└── src/stores/robot.ts         # 支持实时/诊断 WebSocket 连接
```

### 后端职责

- 读取日志目录下的 `.log` 文件。
- 按时间排序并解析日志行。
- 从 `FltStatus: RmstoSendArg` 和 `InfoStatus: ROBOT STATUS` 生成回放帧。
- 解析 `ERRORxxxx` 错误码定义和发生点。
- 构建问题时间线、任务段、噪声折叠结果。
- 从 jarvis-fork 源码补充扫描 `ERRORxxxx` 错误码字典。
- 输出第一版根因候选，并支持根因反馈记录。
- 支持地图别名关系保存，用于低可信地图匹配后的人工确认。
- 支持诊断包导出和导入，包含日志、地图和 manifest。
- 支持缓存统计和清理。
- 模拟 forkweb 已有车端接口：`/api/state`、`/api/map`、`/ws/high`、`/ws/low`。
- 输出 Markdown 和 JSON 报告。

### 前端职责

- 提供 `/replay` 日志诊断页面。
- 配置日志目录、地图目录、手动地图文件。
- 展示概览、地图回放、Top 问题、时间线、错误码中心、任务视角、原始日志过滤。
- 复用 `CanvasView` 显示地图和车辆位置。
- 提供播放、暂停、倍速、进度条拖动跳转。
- 提供诊断包导入/导出、缓存查看/清理、根因反馈和日志上下文复制。
- 提供地图别名管理、时间线高级筛选、错误码筛选、折叠日志分页和缓存分类管理。

## 已知问题

### P0：会影响诊断结论可信度

- 当前样例日志检测到地图名 `3.json`，但本地地图目录只回退加载到 `JA.json`。工具已经提示低可信地图匹配，后续需要拿到现场实际 `3.json` 或建立地图别名关系，否则回放位置可能和现场地图不一致。
- 根因候选分析还是规则第一版，只能输出初步“疑似根因 + 证据链 + 置信度”，还需要更多真实故障日志校准规则和置信度。
- 错误码字典主要来自日志内 `error_name,error_str=ERRORxxxx`，后续仍需要从 jarvis-fork 源码或配置中补齐完整错误码字典。

### P1：会影响第一版现场使用体验

- 任务视角仍缺少真实任务日志验证：当前样例日志 `current_task_id` 始终为 `Null`，任务列表为空是合理结果，但任务切分、任务失败前后日志、任务期间轨迹还需要用真实任务日志验证。
- 进度条事件刻度当前只展示最多 200 个关键事件，后续需要在大日志里做聚合/缩放，避免 marker 太密。
- 诊断包导入已新增本地 zip 路径导入；浏览器选择文件仍使用 base64，后续如需远程网页上传大包，可继续补分片上传。
- 端口和运行时配置已有一键启动脚本处理，后续仍可进一步做成桌面应用或系统服务。

### P2：会影响大日志和长期使用

- 会话缓存已有分类统计、手动清理、按类型清理和过期自动清理；后续还没有缓存版本迁移策略。
- 诊断样本还不够丰富：当前主要用 `/home/xbl/Desktop/log-20260710-173240.log` 验证，后续需要覆盖有真实任务 ID、真实任务失败、定位异常、急停、设备离线、地图加载失败等典型日志。
