# 日志诊断工具产品化方案

本文记录日志诊断与回放工具后续产品化方向，当前不代表已经完成桌面应用或车端部署。

## 桌面应用方案

- 目标：封装 Node、forkweb 前端和 replay-server，让现场人员打开即用。
- 候选技术：Electron 或 Tauri。
- 第一版边界：
  - 启动本地 replay-server。
  - 启动或加载前端页面。
  - 提供日志目录、地图目录、诊断包路径选择。
  - 展示当前版本、端口、缓存目录。
- 风险：
  - 需要维护打包和升级流程。
  - 不同现场系统的权限、杀毒软件和离线依赖需要验证。

## 车端轻量部署方案

- 目标：车端只读收集日志、地图和版本信息，一键生成诊断包。
- 第一版边界：
  - 只读访问日志目录和地图目录。
  - 限制 CPU、IO 和并发。
  - 默认不开放公网访问。
  - 不影响实时控制服务。
- 风险：
  - 车端 Node 环境不一定稳定。
  - 日志目录、地图目录和权限需要按车型确认。
  - 需要明确安全边界和访问认证。

## 一键采集推荐流程

1. 现场人员确认问题时间范围。
2. 执行采集脚本，传入日志目录、地图目录、输出目录。
3. 脚本生成标准目录和 zip 包。
4. 在 forkweb `/replay` 页面使用诊断包导入或路径导入。
5. 导出报告并补充人工结论。
