# forkweb 产品关键决策记录

本文记录 forkweb 从“叉车单机监控与配置前端”调整为“工单驱动的自助诊断系统”过程中确认的 50 个关键产品决策。

## 背景结论

forkweb 当前已有 Vue3 + Vite + Element Plus + Pinia 前端，以及本地 replay-server 日志诊断服务。项目已经具备工单、用户、现场、日志诊断、报告导出、诊断包、错误码、知识库和回放能力。

新的产品主线不是连接车辆和实时控制，而是让现场人员通过问题工单上传日志或诊断包，系统自动分析并生成基于已验证知识库的排查向导。现场人员优先自助解决，解决不了再升级研发。研发处理后继续沉淀知识库，形成闭环。

## 关键决策

1. 产品定位：选择“机器人/叉车问题诊断与现场工单系统”，不再以实时控制台作为主线。

2. 一号用户：选择现场运维/实施/售后人员。他们的需求优先于研发深度调试和客户管理视角。

3. 主线任务：选择“先把问题定位清楚”，暂时没有连接车辆的需求。

4. 第一成功指标：现场人员能通过系统提问题工单，并自主解决一部分问题，解决不了的再找研发。

5. 自助解决层级：选择“问题分类 + 推荐检查步骤”，由系统给出可执行排查路径，现场人员按步骤确认。

6. 排查向导入口：选择从工单开始。工单是日志、诊断、排查、升级和复盘的主容器。

7. 新建工单输入量：选择极简输入，包括标题、现场、问题描述、日志或诊断包。

8. 自动分析完成后的第一屏：第一屏必须优先展示排查向导，同时保留证据仪表盘和 AI/自动结论。

9. 工单详情页层级：顶部组合展示排查向导和结论摘要，下方展示证据仪表盘。

10. 排查步骤来源：规则和知识库生成，AI 只做解释补充，不作为步骤来源。

11. 排查步骤粒度：选择简短检查清单。每一步应能被现场人员执行、勾选和记录结果。

12. 步骤执行结果：每步只记录未检查、已通过、未通过、不适用四种状态。

13. 升级研发触发：关键步骤走完仍未解决，或任一步出现“未通过且无现场可处理动作”时，系统建议升级研发。

14. 确认已解决：需要选择解决方式分类，可选补充一句话。

15. 知识库维护权：第一版只有研发可以新增或修改知识规则。

16. 规则发布门槛：知识规则分为草稿、已验证、已停用等状态，只有已验证规则进入现场排查向导。

17. 工单状态流：围绕现场自助排查重新命名和组织状态，包括待分析、分析中、待现场排查、现场排查中、已自助解决、待研发介入、研发处理中、已解决。

18. replay 定位：`/replay` 不再作为现场主入口，而是工单详情中的证据和回放能力。

19. 登录默认页：现场人员登录后默认进入工单列表。

20. 工单列表筛选：状态是最重要筛选维度，同时需要现场/项目和问题类型筛选。

21. 问题类型来源：系统根据日志和知识库自动分类，用户可以手动修正。

22. 步骤默认内容：默认展示步骤标题、操作说明、判断标准；关联证据放入可展开详情。

23. 多个疑似问题：展示 Top 3 排查路径，默认展开最高优先级路径。

24. Top 3 排序：综合排序，优先级为严重程度、是否阻塞任务或安全、置信度、是否接近用户描述的发生时间。

25. 发生时间：新建工单支持可选填写发生时间或时间范围，用于提升诊断排序和日志定位准确度。

26. 解决方式分类：第一版采用少量通用分类，包括重启恢复、重新接线/供电恢复、更换硬件、调整配置/地图、补充/更换日志后确认、误报/无需处理、其他。

27. 升级研发补充信息：升级时要求选择升级原因，系统自动附带日志、报告、排查步骤状态和诊断包。

28. 知识沉淀触发：研发标记已解决时，系统提示是否沉淀为知识规则。

29. 知识规则生成状态：从研发解决方案生成知识规则时，默认生成草稿，研发补充匹配条件和排查步骤后才能标记已验证。

30. 实时控制/配置能力：短期保留，但从现场人员主导航隐藏或降级到高级/研发入口。

31. 角色导航：售后/现场人员看到工单列表、新建工单、我的待排查；研发看到待研发工单、知识库、诊断工具；管理员看到用户管理和现场管理。

32. 现场人员知识库入口：现场人员不单独看到知识库入口，只通过工单排查向导消费知识。

33. AI 第一版定位：AI 只作为解释器，把规则证据、日志上下文和错误码含义翻译成更易懂的中文说明。

34. 步骤审计信息：记录每一步的处理人和时间戳，但前端不强展示复杂统计。

35. 多次重新分析：同一个工单支持多次重新分析，并保留历史分析版本。

36. 报告和诊断包绑定：报告、诊断包、Top 问题、排查路径和证据快照绑定到每次分析版本。

37. 分析版本差异：多次分析之间做简要差异对比，包括问题类型、Top 3、错误码数量、任务数量、地图匹配、报告生成时间。

38. 工单证据面板：工单详情嵌入精简证据面板，并保留打开高级诊断入口。

39. 高级诊断入口权限：只有研发和管理员可见。

40. 步骤类型：每个排查步骤标记为只读检查、现场操作或需研发处理。

41. 现场操作安全确认：每张工单或每次排查会话首次执行现场操作步骤前，需要安全确认。

42. 跳过步骤：支持“不适用”，但必须选择原因，如现场无该传感器、无权限操作、工具不可用、已由客户确认、其他。

43. 暂停和继续：排查向导支持暂停/稍后继续，工单保持现场排查中，步骤状态自动保存。

44. 预计耗时：排查向导展示粗粒度预计耗时，如约 3-5 分钟、约 10 分钟、需要较长时间。

45. 向导有效性反馈：现场人员解决工单后，对排查向导反馈有用、没用或部分有用。

46. 无效反馈处理：多次“没用”反馈后，知识规则标记为需要研发复查，但不自动停用。

47. 未匹配规则处理：没有匹配到已验证知识规则时，系统显示未匹配到可用排查规则，引导补充材料或升级研发，同时展示基础证据摘要。

48. 影响程度：新建工单支持可选影响程度/紧急程度，如低、中、高、停线。

49. 联系人信息：第一版不在工单里单独填写客户或现场联系人，只依赖项目现场和提单人。

50. 第一版产品形态：选择工单驱动的自助诊断系统。现有回放能力作为证据引擎嵌入工单。

## 第二轮关键决策（部署与运营）

51. 部署场景：上云部署给多个客户现场使用，四五个售后人员，用户量小。

52. 数据隔离：第一版只有内部人员（售后 + 研发 + 管理员）使用，客户不登录系统。售后看自己的工单，知识库全局共享，不按客户/机型区分。

53. 存储策略：上传文件总大小限制 200MB。分析完成后原始日志文件保留 7 天，超期自动删除。报告和数据库长期保留。

54. 通知机制：接入企业微信 Webhook 机器人。在分析完成、升级研发、研发解决等关键节点发送通知到企业微信群。

55. 知识库冷启动：上线前组织研发团队做集中知识库预填充，确保常见问题类型有已验证的排查规则。

56. 文件上传体验：新建工单页面明确展示支持的文件格式说明。分析失败时工单状态不卡在"分析中"，应回退并显示失败原因。

57. 真实日志测试：上线前使用真实叉车现场日志做端到端测试，验证完整流程可用。

58. 移动端适配：售后用笔记本电脑操作，第一版不做移动端适配。

59. 日志获取方式：售后通过 SSH 脚本从叉车拉日志到笔记本，再上传到系统。

60. 并发分析：不加分析任务队列，预期同一时刻只有一个分析任务运行。

61. 部署方式：Docker 容器部署。一键更新脚本自动拉取最新代码、重新编译、重启。更新前自动备份数据库。

62. HTTPS 和域名：使用 sslip.io 域名 + HTTPS。服务器在美国。

63. 账号管理：默认管理员账号 admin/admin123，上线前必须修改默认密码。管理员手动创建账号和重置密码。JWT 有效期 7 天。

64. AI 成本：第一版不限制 AI 调用月度成本上限。

65. 工单关闭与重开：已解决的工单不能重开，问题复发新建工单。新增"已取消"状态用于废弃建错的工单。

66. 数据统计：工单维度（数量、自助解决率、平均解决耗时、现场问题分布）；知识库维度（规则匹配频率、反馈分布、覆盖率）；人员维度（提单量、解决量）。

67. 工单编辑：工单未终结前允许编辑基本信息（标题、描述、现场、影响程度、发生时间），编辑记录到事件流。已解决/已取消的工单不允许编辑。

68. 工单评论：工单详情支持纯文字评论，任何人都能发表，按时间排列混在事件流中展示。

69. 补充上传：工单详情支持追加上传日志文件，上传后可选触发重新分析。

70. 列表分页与排序：工单列表每页 20 条，按创建时间倒序排列。

71. 系统监控：Docker 配置自动重启。心跳脚本定期检查服务健康，失败发企业微信告警。分析任务设 10 分钟超时，超时自动标记分析失败。

72. 上线节奏：先让一两个售后试用一周，收集反馈修一轮后全面推开。写一页纸操作指南发企业微信群。

## 第三轮关键决策（车型管理）

73. 车型层级结构：两级——车型类别（叉车、差速车、双舵轮车、清洁机器人等）+ 具体型号（P1500、A1500、R1500、E2000 等）。

74. 车型数据管理：管理员和研发可在系统中动态新增/编辑车型类别和具体型号，不写死在代码里。

75. 现场关联车型：一个现场可关联多种具体型号（多对多），关联到型号级别而非类别级别。

76. 工单关联车型：一个工单只关联一种具体型号。

77. 工单车型必填：新建工单时车型为必填字段。

78. 工单现场必填：新建工单时现场改为必填字段（因为车型列表依赖现场选择）。

79. 车型联动选择：提工单选车型时，只展示当前所选现场关联的型号，缩小选择范围。

80. 车型管理入口：独立"车型管理"页面，研发和管理员可操作。页面分两级展示，左侧类别列表，右侧选中类别下的型号列表。

81. 知识库车型关联：第一版不做，知识规则暂不限制适用车型，留到后续迭代。

82. 工单列表车型筛选：工单列表增加按车型筛选。

83. 统计车型维度：统计仪表盘增加按车型分布的统计。

84. 车型删除保护：已被现场或工单关联的车型类别/型号不允许删除。

85. 知识库车型类别绑定粒度：知识规则绑定到车型**类别**（如叉车、差速车），而非具体型号。

86. 知识规则类别绑定关系：一条规则可绑定多个类别（多选）；不绑定任何类别的规则为全局通用规则。

87. 知识库匹配逻辑：工单有车型时，匹配规则 = 该车型类别的规则 + 通用规则；工单无车型（老工单）时匹配全部规则。

88. 知识规则编辑表单：在知识规则编辑表单中增加"适用车型类别"多选字段，不选即为通用规则。

89. 从工单沉淀知识时预填类别：从工单创建知识规则时，根据工单车型自动预填该车型的类别，但允许修改。

90. 知识规则存储方式：在 KnowledgeRule JSON 对象中直接存储 vehicleCategoryIds: number[]，空数组表示通用规则。

91. 现有规则处理：已存在的 12 条种子规则保持通用（不绑定类别）。

92. 知识库管理筛选：知识库管理对话框增加"车型类别"下拉筛选，支持查看特定类别或通用规则。

93. 知识库导出按类别：导出时可选择导出全部、指定类别的规则，并可选是否包含通用规则。

94. 知识库导入保留类别：导入时 JSON 中的 vehicleCategoryIds 会被保留。

95. 系统品牌名称：对外展示为 "Junion 单机售后工单系统"，简称 "Junion"。

96. 登录页风格：深蓝色工业风渐变背景 + 点阵纹理 + 毛玻璃卡片 + 玖物智能 Logo。

97. 全局顶栏：左侧页面标题，右侧用户名 + 角色标签（售后/研发/管理员）+ 退出按钮。

98. 侧边栏 Logo：favicon 小图标 + "Junion" 文字，不改侧边栏整体风格。

99. 主色调：蓝色系（Element Plus 默认蓝 #2563eb），全局统一。

100. 工单状态颜色：明确区分——待分析灰、分析中蓝、排查相关橙、研发相关红、已解决绿、已取消灰。

101. 统计仪表盘美化：数字卡片加彩色左边框（蓝/绿/橙），小节标题加蓝色竖线装饰。

102. 工单详情美化：section-title 蓝色竖线，AI 结论渐变蓝色背景，版本栏蓝色背景。

103. 全局卡片样式：圆角 10px，表头 #f8fafc 背景色，section-title 统一左侧蓝色竖线。

104. 浏览器标签：标题 "Junion 售后系统"，favicon 用 Logo 左侧圆形图案。

105. 版权信息：登录页底部显示 © 2026 Junion。

106. 移动端适配：不做。

107. 过渡动画：不做。

## 产品闭环

1. 现场人员登录后进入工单列表。
2. 现场人员极简创建工单，上传日志或诊断包。
3. 系统创建分析版本，解析日志、地图、错误码、任务、根因候选和知识库匹配。
4. 工单详情顶部展示结论摘要和 Top 3 排查路径。
5. 现场人员按步骤排查，系统保存步骤状态、处理人和时间戳。
6. 可自助解决时，现场人员选择解决分类并评价向导有效性。
7. 无法自助解决时，系统建议升级研发，并附带诊断包、报告、步骤状态和升级原因。
8. 研发处理后标记解决，并可生成知识规则草稿。
9. 研发验证规则后发布到现场排查向导。
