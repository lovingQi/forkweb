# 叉车日志诊断与回放工具使用说明书

本文面向现场测试、部署和研发排查人员，说明 forkweb 内置的日志诊断与回放工具如何使用。工具入口为 forkweb 的 `/replay` 页面，核心目标是把离线日志、地图、任务、错误码和原始日志上下文放到同一个页面里，减少用记事本逐行查日志的时间。

## 1. 工具定位

### 1.1 适用场景

这个工具适合处理以下问题：

- 现场人员部署或测试时，中途出现异常，需要快速知道叉车在什么时间、什么位置、执行什么任务、发生了什么错误。
- 当前没有连接实车，但已经从车上下载了日志和地图，需要在本机离线诊断。
- 需要把一次现场问题整理成诊断包，方便研发复现、回放、查看地图和日志。
- 需要导出 Markdown 或 JSON 报告，用于问题流转、复盘、归档。
- 需要排查 `ERRORxxxx` 错误码、关键 `[E]` 日志、关键 `[W]` 日志、低定位分、急停、丢失、任务失败等问题。

### 1.2 当前第一版边界

当前版本以本地离线诊断优先：

- 支持读取本机日志目录下的 `.log` 文件。
- 支持读取本机地图目录或手动指定地图 JSON 文件。
- 支持导入已经打包好的诊断包，在无法连接实车时也能诊断。
- 支持复用 forkweb 地图显示能力，在 `/replay` 页面中回放车辆轨迹。
- 支持导出报告和诊断包。

当前版本不是实车实时监控页面的替代品。实时监控仍使用 forkweb 原来的页面，日志回放和诊断集中在 `/replay` 页面。

## 2. 进入页面

启动前端和本地诊断服务后，在浏览器打开：

```text
http://127.0.0.1:5173/replay
```

如果使用一键启动脚本，端口可能会自动变化，终端会打印实际地址。例如当前验证时可能是：

```text
http://127.0.0.1:5174/replay
```

进入页面后，顶部会看到日志目录、地图目录、地图文件等配置项，以及加载、重新解析、导入诊断包、导出诊断包、地图别名、缓存等操作按钮。

## 3. 加载日志和地图

### 3.1 日志目录

日志目录用于指定 `.log` 文件所在目录。默认值通常为：

```text
/home/xbl/Desktop
```

页面加载时会读取该目录下的 `.log` 文件，例如：

```text
/home/xbl/Desktop/log-20260710-173240.log
```

使用建议：

- 一次现场问题最好把相关日志放在同一个目录。
- 如果目录里有多份 `.log`，工具会一起读取并按时间排序。
- 如果只想诊断某一次问题，建议把不相关日志临时移到其他目录，避免时间线和统计被无关日志干扰。

### 3.2 地图目录

地图目录用于让工具自动寻找地图 JSON。默认值通常为：

```text
/home/xbl/Desktop/jarvis-fork/params/map
```

工具会根据日志中识别到的地图名、地图目录里的 JSON 文件名、已保存的地图别名关系来自动选择地图。

### 3.3 手动地图文件

如果自动匹配的地图不对，或者地图目录里有多个相似地图，可以在“地图文件”里填写具体 JSON 文件的完整路径。

示例：

```text
/home/xbl/Desktop/jarvis-fork/params/map/site_a/main_map.json
```

手动地图文件优先级最高。现场排查时，如果已经知道车上当时使用的是哪张地图，建议直接填手动地图文件，减少自动匹配误差。

### 3.4 加载和重新解析

常用按钮含义：

- “加载”或“诊断”：按当前日志目录、地图目录、地图文件配置读取数据。
- “重新解析”：忽略已有会话缓存，重新读取日志和地图。日志或地图文件改动后建议使用这个入口。
- “导入诊断包”：直接读取别人导出的 zip 诊断包，适合离线复现。
- “路径导入”：在诊断包输入框填写本机 zip 完整路径后导入，适合大诊断包，避免浏览器一次性读取大文件。

## 4. 一键概览

加载完成后，页面上方会显示一键概览。它用于快速判断这批日志有没有可诊断价值，以及问题大致集中在哪里。

### 4.1 基础信息

概览中通常包含：

- 日志时间范围：日志中最早和最晚的时间。
- 日志文件数量：本次读取了多少个 `.log` 文件。
- 回放帧数：可用于地图回放的位置状态数量。
- 任务数量：解析出的任务段数量。
- 错误事件数：错误、警告、错误码等事件数量。
- 错误码数量：识别到的 `ERRORxxxx` 种类数量。
- 车辆名：从日志中解析出的车名或机器人标识。
- 地图名：日志中识别到的地图名或最终匹配的地图文件。
- 分支、版本、提交信息：如果日志中包含相关字段，会在概览中展示。

### 4.2 数据完整性

数据完整性用于判断当前诊断是否可信。重点看这些标签：

- 是否有地图：没有地图时仍可看日志、错误码、时间线，但地图回放能力受限。
- 是否有可回放帧：没有帧时说明日志里缺少有效位置状态，无法进行轨迹回放。
- 是否有任务 ID：没有任务 ID 时任务视角会比较弱。
- 是否有错误码定义：没有定义时仍能识别 `ERRORxxxx` 发生点，但错误描述可能不完整。

现场使用建议：

- 如果“没有可回放帧”，优先检查日志目录是否选错，或日志是否缺少 `FltStatus: RmstoSendArg` / `InfoStatus: ROBOT STATUS`。
- 如果“没有地图”，优先检查地图目录或手动地图文件。
- 如果“没有错误码定义”，排查仍可继续，但需要结合原始日志上下文看含义。

### 4.3 地图匹配置信度

地图匹配信息通常包含：

- 匹配策略：手动指定、别名匹配、精确匹配、包含匹配、默认第一张地图等。
- 置信度：数值越高说明工具越确信当前地图就是日志对应地图。
- 日志地图名：从日志中识别出的地图名。
- 选中地图文件：实际用于回放的地图 JSON。

使用建议：

- 置信度高且地图显示正常，可以直接继续看回放。
- 置信度低时，先确认地图是否正确。
- 如果确认当前地图正确，可以使用“确认地图”或地图别名功能保存关系，后续同类日志会自动匹配。
- 如果地图明显不对，手动填写地图文件后重新解析。

### 4.4 Top 问题列表

Top 问题会汇总最值得先看的异常，例如：

- 高等级错误码。
- 关键 `[E]` 日志。
- 关键 `[W]` 日志。
- 任务失败候选。
- 定位分异常。
- 急停、lost、Stopped 等状态变化。

点击 Top 问题后，页面会跳转到对应时间点，地图回放、时间线和原始日志上下文会围绕该时间点联动。

## 5. 地图回放

地图回放区域用于回答三个问题：

- 叉车什么时候在什么位置。
- 当时处于什么状态，执行什么任务。
- 轨迹附近发生了哪些错误或关键事件。

### 5.1 数据来源

当前版本主要使用：

- `FltStatus: RmstoSendArg` 中的 `x/y/theta` 作为主位置来源。
- `InfoStatus: ROBOT STATUS` 中的 `loc(pose...)` 作为辅助补充。
- 日志中的任务 ID、状态、错误码、定位分、电量、速度、货叉高度等作为状态信息。

### 5.2 播放和暂停

常用操作：

- 点击播放按钮开始按时间推进。
- 点击暂停按钮停在当前时刻。
- 回放过程中，当前车辆位置、状态、任务、错误信息会随时间更新。

### 5.3 倍速

倍速用于控制回放快慢。现场排查建议：

- 先用较快倍速浏览整体过程。
- 接近错误发生时间后暂停或降低倍速。
- 对任务失败、急停、定位分下降等问题，建议在异常前后慢速查看。

### 5.4 视频式进度条

进度条表现类似视频播放器：

- 显示当前回放进度。
- 显示当前时间和总时长。
- 拖动进度条可跳转到任意日志时刻。
- 回放中进度条会自动跟随后端播放状态推进。

使用建议：

- 先点击 Top 问题或时间线事件跳到异常附近。
- 再拖动进度条查看异常前后的轨迹变化。
- 如果日志时间跨度很长，直接拖动比从头播放效率更高。

### 5.5 事件 marker

进度条上会显示事件刻度 marker。marker 表示该时间附近有关键事件，例如错误码、任务变化、关键警告等。

操作方式：

- 点击 marker 可以跳转到对应时间。
- 如果同一小段时间内有多个事件，marker 可能显示数量。
- marker 颜色用于区分错误、警告、普通信息等严重程度。

### 5.6 轨迹线和事件点

地图上会显示：

- 当前车位置。
- 历史轨迹线。
- 错误或任务事件点。

操作方式：

- 点击事件点：跳转到事件发生时间，并查看事件详情。
- 点击轨迹点：查看该时刻的位置、状态、任务、错误等信息。
- 开启或关闭轨迹平滑：用于观察轨迹趋势或保留原始采样效果。

### 5.7 回放模式

当前支持两种回放模式：

- 真实时间模式：尽量按日志真实时间间隔播放，适合复盘完整过程。
- 按状态帧模式：按解析出的状态帧推进，适合快速查看每个有效状态点。

如果日志中状态帧间隔不均匀，按状态帧模式会更容易快速浏览。

## 6. 诊断结论和根因证据

诊断结论区域用于快速给出候选根因，而不是只罗列日志。

### 6.1 根因候选

根因候选通常包含：

- 疑似根因标题。
- 置信度。
- 关联错误码。
- 关联任务。
- 证据列表。
- 处理建议。

这些结论来自当前内置规则和日志证据聚合。它可以帮助快速定位方向，但最终仍需要结合现场情况确认。

### 6.2 证据查看

每条根因候选会尽量关联到证据，例如：

- 发生时间。
- 原始日志片段。
- 错误码发生点。
- 任务失败前后上下文。
- 状态变化或定位分变化。

点击相关证据后，可以跳转地图时间或查看原始日志上下文。

### 6.3 有用和误报反馈

根因候选旁边提供“有用”或“误报”反馈入口。用途是记录当前规则是否对现场排查有帮助。

使用建议：

- 如果候选根因确实帮你定位了问题，标记“有用”。
- 如果候选根因明显不是本次问题原因，标记“误报”。
- 反馈会写入本地缓存，后续可用于优化根因规则。

## 7. 问题时间线

问题时间线按时间顺序展示关键事件，用于快速串起现场问题发生过程。

### 7.1 事件类型

当前第一版重点展示：

- 系统启动。
- Web 或服务启动。
- 机器人连接成功。
- 地图或参数加载异常。
- 任务开始、任务变化。
- 错误码发生。
- `[E]` 级日志。
- 关键 `[W]` 日志。
- 状态变化，例如 Running、Stopped、lost、estop。
- 低定位分事件。

### 7.2 筛选

时间线支持多种筛选：

- 按事件类型筛选。
- 按时间范围筛选。
- 按严重度筛选。
- 只看真实故障。
- 只看配置提醒。
- 是否显示噪声。
- 是否按严重度排序。

现场建议：

- 不知道问题原因时，先看全部关键事件。
- 现场报错明确时，先筛选错误码或错误级别。
- 日志很吵时，打开噪声折叠或只看关键事件。

### 7.3 重复合并

如果同一类事件在短时间内重复出现，可以开启重复合并。合并后时间线更短，更适合快速看主线。

使用建议：

- 高频重复日志很多时开启。
- 需要确认每一次重复发生时间时关闭。

### 7.4 点击联动

点击时间线事件后，会联动：

- 地图回放跳转到该时间。
- 当前状态更新为该时刻附近状态。
- 原始日志上下文刷新到该事件前后。
- 页面自动切换到“原始日志/过滤”Tab。
- 原始日志表中会高亮与事件关联的日志行。

这是现场排查最常用的入口之一。建议按“时间线事件 -> 地图位置 -> 原始日志上下文”的顺序看。

## 8. 错误码中心

错误码中心用于集中查看 `ERRORxxxx`。

### 8.1 解析来源

当前会从以下位置识别错误码：

- `error_name,error_str=ERRORxxxx,{...}` 错误码定义。
- `current_code is ERRORxxxx`。
- `current_task_error_code is ERRORxxxx`。
- `RmstoSendArg.errors`。
- 日志正文中的 `ERRORxxxx`。
- jarvis-fork 源码中可扫描到的 `ERRORxxxx` 定义或引用。

### 8.2 列表字段

错误码中心通常展示：

- 错误码。
- 描述。
- 屏幕提示。
- 等级。
- 类型，例如真实故障、配置提醒、定义、未知文本。
- 发生次数。
- 首次发生时间。
- 末次发生时间。
- 关联任务。
- 来源和置信度。

### 8.3 筛选

支持按以下维度筛选：

- 错误码关键词。
- 模块。
- 等级。
- 类型。

使用建议：

- 已知错误码时，直接输入 `ERRORxxxx` 搜索。
- 只想看影响现场问题的错误时，优先筛选真实故障。
- 想整理错误码字典时，可以关注定义类和未知文本类。

### 8.4 发生点

点击某个错误码后，会展示所有发生点。发生点通常包含：

- 发生时间。
- 关联任务。
- 原始日志摘要。
- 发生来源。

点击发生点可以跳转地图回放时间，也可以查看原始日志上下文。

如果错误码发生点本身没有显式任务 ID，工具会尝试按发生时间匹配当时所在任务段，并在错误码中心中展示关联任务。

## 9. 任务视角

任务视角用于从任务角度理解问题，尤其适合排查“某个任务为什么失败”。

### 9.1 解析字段

当前重点解析：

- `current_task_id`。
- `last_finished_task_id`。
- `last_finished_task_is_success`。
- `unfinished_path`。
- `new_unfinished_path`。
- `status`。
- `FltTask` 相关任务日志。
- 日志中可解析到的 `current_routes`。

### 9.2 任务列表

每个任务通常展示：

- 任务 ID。
- 开始时间。
- 结束时间。
- 任务状态。
- 完成任务 ID。
- 成功或失败标记。
- 未完成路径。
- 关联错误码。
- 任务期间轨迹范围。

### 9.3 任务失败排查

建议流程：

1. 在任务列表中找到失败或异常任务。
2. 点击任务，让时间线、错误码、轨迹和原始日志过滤到该任务相关范围。
3. 查看任务失败前后日志上下文。
4. 结合地图轨迹判断是路径、定位、任务参数、硬件状态还是调度问题。

## 10. 原始日志和过滤

原始日志区域用于在自动分析之外保留可追溯证据。

### 10.1 过滤条件

支持按以下条件过滤：

- 日志级别 `[E]`、`[W]`、`[I]`、`[D]`。
- 模块名。
- 关键词。
- 错误码。
- 任务 ID。
- 开始时间。
- 结束时间。
- 是否噪声。
- 是否关键事件。

使用建议：

- 先通过时间线或错误码跳到关键时间。
- 再用当前时间上下文看前后日志。
- 如果需要扩大范围，再调整开始和结束时间。

### 10.2 当前时间上下文

当前时间上下文会围绕回放当前时间展示前后日志片段。适合回答：

- 异常发生前系统在做什么。
- 异常发生后状态如何变化。
- 同一时间是否还有其他模块报错。

### 10.3 复制上下文

支持复制：

- 当前页日志。
- 当前时间上下文。
- 折叠日志详情当前页。

现场反馈问题时，建议复制异常前后上下文，而不是只复制一行错误。这样研发更容易判断因果关系。

### 10.4 噪声折叠详情

工具会把部分高频、重复、低价值日志折叠起来，例如：

- 高频 `sent: toSendV`。
- 高频 `FltTask: start/end calc current task id`。
- 高频 `unfinshed path ... null`。
- 高频空值 `Invalid task error code`。
- 重复 `Get charging error state is: 0`。
- 重复 `get battery failed`。
- 启动阶段参数缺省类配置提醒。

点击折叠噪声标签可以分页查看完整被折叠日志。需要确认某类重复日志是否真的影响问题时，可以打开详情查看。

## 11. 地图别名管理

地图别名用于保存“日志里识别到的地图名”和“本机实际地图文件”的关系。

### 11.1 什么时候需要地图别名

以下情况建议保存地图别名：

- 日志地图名和本机地图文件名不完全一致。
- 自动匹配置信度低，但你已经确认地图正确。
- 同一现场经常使用同一张地图，希望后续自动匹配。

### 11.2 确认地图

当页面提示低可信地图匹配时，如果确认当前选中地图正确，可以点击确认地图。确认后工具会保存别名关系。

保存后，后续相同日志地图名会优先通过别名匹配。

### 11.3 管理地图别名

点击“地图别名”可以打开管理弹窗。当前支持：

- 查看已保存别名。
- 删除别名。
- 导入别名 JSON。
- 导出别名 JSON。
- 导入时选择是否覆盖同名别名。
- 检查别名冲突。

如果导入诊断包时发现包内别名和本地别名冲突，工具不会静默覆盖，需要到地图别名管理中人工确认。

## 12. 诊断包和报告

### 12.1 导出诊断包

诊断包是 zip 文件，适合把一次现场问题整体交给其他人复现。

当前导出内容包括：

- 本次使用的日志。
- 匹配到的地图文件。
- `diagnostic-package.json` manifest。
- `reports/report.md`。
- `reports/report.json`。
- `config/map-alias.json`。
- 根因反馈摘要。

导出建议：

- 现场问题需要研发协助时，优先导出诊断包。
- 诊断包比单独发日志更完整，因为包含地图、报告和匹配信息。

### 12.2 导入诊断包

点击“导入诊断包”，选择别人导出的 zip。导入后工具会：

- 解压到本地缓存目录。
- 自动识别包内日志目录。
- 自动识别包内地图文件。
- 返回包信息。
- 返回包内地图别名和冲突提示。
- 使用包内日志和地图重新加载诊断。

导入后不需要连接实车，也不需要本机已有同名地图。

如果诊断包比较大，推荐使用“路径导入”：

1. 将 zip 放到本机目录。
2. 在页面顶部“诊断包”输入框填写 zip 完整路径。
3. 点击“路径导入”。

路径导入由本地诊断服务直接读取文件，不需要浏览器把整个 zip 转成 base64 后再提交，适合大日志包。

### 12.3 Markdown 报告

Markdown 报告适合人工阅读、贴到问题单或复盘文档中。内容包括：

- 诊断结论。
- 地图匹配。
- 根因候选。
- 错误码分组。
- 任务摘要。
- 文件索引。
- 关键时间线摘要。

### 12.4 JSON 报告

JSON 报告适合程序处理和二次分析。内容结构化，包含：

- 概览数据。
- 根因候选。
- 地图匹配信息。
- 错误码聚合。
- 任务摘要。
- 关键时间线。
- 诊断文件索引。

## 13. 缓存管理

工具会使用本地缓存提高重复解析速度，并保存导入、导出和反馈信息。

### 13.1 缓存详情

点击“缓存”可以查看：

- 缓存版本。
- 文件数量。
- 总大小。
- 分类统计。

当前分类包括：

- 会话缓存。
- 导出诊断包。
- 导入诊断包。
- 根因反馈。
- 其他缓存。

### 13.2 清理缓存

支持：

- 清理全部缓存。
- 按分类清理缓存。
- 自动清理过期缓存。

使用建议：

- 日志或地图改动后，如果页面仍显示旧数据，可以先“重新解析”，再视情况清理会话缓存。
- 诊断包很多导致空间占用较大时，可以清理导出诊断包或导入诊断包缓存。
- 清理缓存不会删除原始日志目录和原始地图目录里的文件。

## 14. 推荐现场排查流程

### 14.1 本机已有日志和地图

1. 打开 `/replay` 页面。
2. 填写日志目录。
3. 填写地图目录，必要时填写手动地图文件。
4. 点击加载或重新解析。
5. 先看一键概览的数据完整性和 Top 问题。
6. 点击 Top 问题或时间线事件跳转到异常时间。
7. 在地图回放中查看车辆位置、轨迹、任务、错误。
8. 在原始日志上下文中复制异常前后证据。
9. 在错误码中心查看关联 `ERRORxxxx` 的定义、等级和发生点。
10. 如需转交研发，导出诊断包和 Markdown 报告。

### 14.2 只有诊断包

1. 打开 `/replay` 页面。
2. 点击导入诊断包。
3. 选择 zip 文件。
4. 等待页面自动加载包内日志和地图。
5. 查看诊断包信息和别名冲突提示。
6. 按概览、地图回放、时间线、错误码中心、原始日志的顺序排查。

如果 zip 较大，可以改用“路径导入”，填写本地 zip 完整路径后加载。

### 14.3 地图匹配不确定

1. 查看概览中的地图匹配策略和置信度。
2. 如果置信度低，先观察地图显示是否和轨迹明显对应。
3. 如果地图不对，填写手动地图文件并重新解析。
4. 如果地图正确，点击确认地图保存别名。
5. 后续相同现场日志可自动使用别名匹配。

## 15. 常见使用误区

- 只看错误码，不看时间线：错误码可能是结果，不一定是根因，需要看异常前后的事件顺序。
- 只看一行 `[E]` 日志：单行日志缺少上下文，建议复制前后日志片段。
- 地图低可信但直接判断轨迹异常：先确认地图是否匹配，否则位置判断可能失真。
- 日志目录放太多无关日志：会让统计、时间线和 Top 问题混在一起。
- 导入别名时直接覆盖：如果本机和诊断包的地图路径不同，覆盖前要确认冲突。

## 16. 当前限制

- 回放效果依赖日志中是否存在有效 `FltStatus` 或 `InfoStatus`。
- 自动根因是候选结论，不能替代人工最终判断。
- 地图自动匹配依赖日志地图名、地图文件名和别名关系，低可信时需要人工确认。
- 任务视角依赖日志中任务 ID 和任务状态字段的完整性。
- 当前以本地离线诊断为主，车端部署是后续预留方向。

## 17. 新增增强功能使用补充

### 17.1 异步加载

日志很大时可以点击“异步加载”。页面会显示当前阶段和百分比，例如读取日志、解析状态帧、构建时间线等。加载完成后会自动刷新概览、地图、时间线和原始日志。

### 17.2 健康分和建议关注时间

概览区新增：

- 健康分：综合地图、回放帧、任务、错误码定义、错误数量等信息。
- 日志质量分：用于判断这批日志是否足够完整。
- 建议关注时间：工具认为最值得优先查看的几个时间点，点击即可跳转地图回放。
- 解析耗时和缓存命中：用于判断本次加载是走缓存还是重新解析。

### 17.3 根因解释和证据包

诊断结论中新增：

- 触发规则。
- 正向证据。
- 反向证据。
- 置信因素。
- 复制证据包。
- 加书签。

现场转交问题时，建议先点击“复制证据包”，再补充人工结论，这样研发能更快看到工具判断依据。

### 17.4 人工书签

可以从以下位置添加书签：

- 诊断结论。
- 时间线事件。
- 错误码发生点。
- 当前回放时刻。

书签会显示在地图右侧，并可点击跳转。导出的 Markdown 报告和诊断包会保留书签，适合现场人员把“我认为重要的时刻”交给研发继续分析。

### 17.5 人工结论

点击“人工结论”可以填写：

- 现场。
- 车辆。
- 测试人员。
- 测试轮次。
- 处理状态。
- 已确认根因。
- 备注。

这些信息会进入报告和诊断包 manifest。建议每次准备转交问题前都填写一次。

### 17.6 回放增强

地图回放新增：

- 上一问题、下一问题：快速在异常点之间跳转。
- 循环：配合“设起点”“设终点”反复播放某一段。
- 遇问题暂停：播放到错误、急停、lost、低定位分等异常附近自动暂停。
- 截图：当前版本会复制当前回放时刻、位置、状态、任务和错误信息，作为轻量快照。

### 17.7 原始日志多关键词

原始日志新增“多关键词逗号分隔”。示例：

```text
ERROR,task,lost
```

工具会只保留同时命中这些关键词的日志，并高亮关键词。也可以设置“上下文秒”，按当前时间点前后 N 秒筛选日志。

### 17.8 诊断包选项导出

点击“导出诊断包”后可以选择：

- 时间范围。
- 是否包含地图。
- 是否包含报告。
- 是否包含地图别名。
- 是否包含根因反馈。

注意：当前版本已支持参数和 manifest 记录，但日志文件仍按原文件复制，尚未按时间范围裁剪。大日志瘦身还需要后续继续优化。

### 17.9 诊断包对比

点击“诊断包对比”，分别选择两个诊断包中的 `diagnostic-package.json`，可以对比：

- 诊断包版本。
- 车辆名。
- 地图名。
- 错误数量变化。
- 任务数量变化。
- 回放帧数量变化。

第一版适合快速判断两次现场问题是否来自同一车辆、同一地图，以及错误规模是否变化。

## 18. 诊断知识库

诊断知识库用于沉淀研发人员的排查经验。你可以把一次人工分析保存为规则，下次遇到相似日志时，工具会自动命中并输出到诊断结论。

### 18.1 沉淀知识

常用入口：

- 在“原始日志/过滤”中勾选多行日志，点击“选中作为证据”，再点击“沉淀知识”。
- 在“时间线”事件操作中点击“沉淀知识”。
- 在“错误码中心”的发生点操作中点击“沉淀知识”。
- 在“诊断结论”中点击“转为知识”。

弹窗中填写：

- 标题。
- 问题描述。
- 确认根因。
- 处理办法。
- 严重度。
- 标签。
- 匹配规则，例如必选词、任意词、排除词、模块、等级、错误码、时间窗口、次数阈值。

可以点击“自动提取规则”，工具会根据证据日志提取候选模块、关键词、错误码和等级。保存前建议点击“试跑”，确认规则能命中当前日志。

### 18.2 管理知识库

点击顶部“知识库”按钮，可以：

- 查看知识规则。
- 搜索规则。
- 新增规则。
- 编辑规则。
- 复制规则。
- 删除规则。
- 启用或停用规则。
- 导入知识库 JSON。
- 导出知识库 JSON。
- 用当前日志试跑规则。

停用后的规则不会参与自动诊断。

### 18.3 理解知识库命中

如果某条诊断结论来自知识库，页面会显示“知识库命中”标签。报告中也会新增“知识库命中”小节，包含：

- 命中的知识标题。
- 置信度。
- 命中的规则项。
- 证据日志。
- 历史处理办法。

知识库命中仍然需要研发确认。第一版规则是字符串和时间窗口匹配，规则过宽时可能误报。
